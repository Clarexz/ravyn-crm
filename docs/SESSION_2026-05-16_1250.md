## Decisión arquitectónica crítica: RLS bypass

Las políticas RLS de la BD usan `auth.jwt() ->> 'clinic_id'` como custom claim, pero **el JWT emitido al login NO incluye ese claim** (requeriría configurar un Auth Hook en Supabase, fuera de scope inmediato). Esto significa que cualquier query/insert hecho con el cliente de sesión del usuario es bloqueado por RLS — incluso leer su propio perfil en `users`.

**Solución adoptada:** Bypass de RLS usando service role en el server. La aislación entre clínicas se enforza en código.

- `lib/supabase/auth.ts::getClinicContext()` autentica con `auth.getUser()` (sesión), resuelve `clinic_id` desde tabla `users` usando el **admin client**, y devuelve `{ user, profile, clinicId, db, supabase }`.
- `ctx.db` = cliente admin (bypassa RLS). **Todas las API routes y server components usan `ctx.db`** con `.eq("clinic_id", ctx.clinicId)` explícito.
- `ctx.supabase` = cliente de sesión, reservado para `auth.signOut()` u operaciones que requieran identidad del usuario.
- Service role nunca sale del server (`SUPABASE_SERVICE_ROLE_KEY` en `.env.local`, sin prefijo `NEXT_PUBLIC_`).

Si en el futuro se configura el JWT custom claim, migrar a `ctx.supabase` y dejar que RLS haga el trabajo.

## Nueva sección: Servicios

Sección completamente nueva agregada en esta sesión. Permite a cada clínica gestionar su catálogo de servicios con nombre, descripción, costo, color y estado activo/inactivo.

**Cambios en BD (hechos por el usuario en Supabase):**
- Nueva tabla `public.services` con RLS, trigger `set_updated_at`, índice por `clinic_id`, columna `color` con default `#6366f1`
- `appointments.service_id` (FK → `services.id`, ON DELETE SET NULL)
- `appointments.cost_at_booking` (numeric, snapshot del precio al agendar)

**Frontend:**
- `app/api/services/route.ts` (GET con `?active=true`, POST)
- `app/api/services/[id]/route.ts` (PATCH, DELETE)
- `app/(dashboard)/services/page.tsx` + `loading.tsx`
- `components/crm/ServicesClient.tsx` — tabla + dialog create/edit con color picker (20 colores predefinidos en grid + input nativo de color personalizado)
- Sidebar: nueva entrada "Servicios" con icono `Tag`

**Integración con citas:**
- `NewAppointmentModal` y modo edición de cita en `AppointmentsClient` ahora alimentan el dropdown de servicios desde `/api/services?active=true`
- Al crear/editar cita, manda `service` (nombre snapshot), `service_id` (FK), `cost_at_booking` (snapshot del costo actual del servicio)
- En el calendario, el color del bloque viene de `services(color)` joineado en el query. Cancelled/no_show usan gris `#6b7280`. Cancelled lleva `line-through` y opacidad 60%.

## Componentes nuevos importantes

### `TimeSlotPicker.tsx`
Selector inteligente de hora que reemplazó el `<input type="time">` nativo. Horario laboral hardcoded 9:00–18:00, granularidad 30 min.

Lógica adaptativa según ocupación del día:
- **< 50% ocupado:** muestra lista de citas existentes ("Horas ocupadas") + selector con todos los slots libres
- **≥ 50% ocupado:** muestra bloques contiguos de tiempo libre clickables

Filtros aplicados:
- Slots anteriores a la hora actual (si la fecha es hoy) no se muestran
- Citas con `status === "cancelled"` no cuentan como ocupadas
- Prop `excludeId` permite excluir una cita específica (útil al editar — la propia cita no se cuenta como ocupada)

### `Spinner.tsx`
Componente compartido con `Loader2` de lucide. Usado en los `loading.tsx` de cada ruta del dashboard. La latencia entre pestañas tras cold start es un punto sensible para el usuario.

### `loading.tsx` por ruta
Agregados en `dashboard/`, `appointments/`, `patients/`, `reports/`, `settings/`, `services/`. Solo se activan cuando la página destino suspende (server component async).

## Convenciones y patrones consolidados en esta sesión

1. **Espacio reservado para errores de formulario:** en formularios con validación, cada `<p>` de error usa `min-h-[1rem] leading-4` y siempre renderiza un placeholder (` `) cuando no hay error. Evita layout shift cuando aparecen/desaparecen mensajes.

2. **Reset de modales al cerrar:** `onOpenChange` siempre debe llamar a `resetForm()` o equivalente cuando `!open`. El usuario reportó este bug en el dialog de crear paciente.

3. **Click-outside para dropdowns custom:** patrón estándar con `useRef` + `useEffect` con listener `mousedown` global. Usado en `NewAppointmentModal` (search paciente) y `TimeSlotPicker`.

4. **Validación de fecha/hora futuras:** `mode: "onBlur"` + `reValidateMode: "onChange"` en `useForm`. Refinement de zod cross-field con `trigger(["date", "time"])` en el `onBlur` de cada input.

5. **Estado sincronizado con props server-side:** componentes cliente que reciben datos del server (vía `router.refresh()`) deben usar `useEffect(() => setState(prop), [prop])` para resincronizar. Bug encontrado en `AppointmentsClient` — confirmar mismo patrón en futuros clientes.

6. **Edición de status como parte de edición general:** ya no hay UI separada para "cambiar status". El sheet de detalle de cita tiene un botón "Editar" único que abre el formulario completo (fecha, hora, duración, servicio, status, notas).
